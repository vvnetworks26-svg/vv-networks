# Security — VV Networks

## Overview

Security is applied in layers: transport, application, authentication, and data.

---

## Transport Security

| Measure | Implementation |
|---|---|
| HTTPS only | Enforced by Render and Vercel (automatic TLS) |
| HSTS | `Strict-Transport-Security` via Helmet |
| Security headers | Helmet: `X-Content-Type-Options`, `X-Frame-Options`, `X-XSS-Protection` |
| CSP | Enabled in production via Helmet `contentSecurityPolicy` |
| Compression | gzip in production (configured via `COMPRESSION_ENABLED`) |

---

## Authentication

| Measure | Detail |
|---|---|
| JWT Access Tokens | Short-lived (15m), signed with `JWT_SECRET` |
| Refresh Tokens | Long-lived (30d), stored hashed (SHA-256) in MongoDB |
| Token Rotation | Every refresh issues a new pair and revokes the old token |
| Reuse Detection | Refresh token reuse revokes the entire token family |
| bcrypt | Password hashing with configurable work factor (default 12) |
| Rate Limiting | Auth endpoints: 10 req/min per IP |

---

## API Security

| Measure | Detail |
|---|---|
| Global rate limit | 100 req/min per IP (configurable) |
| CORS | Restricted to configured origins in production. Wildcard `*` blocked. |
| Zod validation | All request bodies validated with Zod schemas |
| SQL injection | Not applicable (MongoDB with Mongoose ODM) |
| NoSQL injection | Prevented by Mongoose schema types |
| Request size limit | 100kb body limit |
| Trust proxy | Enabled in production for correct IP rate limiting behind load balancer |

---

## Stripe Webhook Security

All webhook payloads are verified using Stripe's signature mechanism before processing:

```
stripe.webhooks.constructEvent(payload, signature, STRIPE_WEBHOOK_SECRET)
```

Raw body is parsed separately (before `express.json()`) to preserve the signature.
Any request with an invalid or missing signature returns 400 immediately.

---

## Environment Variables

- All secrets are stored in Render environment variables, never in source code
- `.env` files are in `.gitignore` — never committed
- `JWT_SECRET` is auto-generated by Render (`generateValue: true`)
- Startup validator blocks production launch if required secrets are missing or use default values
- Sensitive fields are automatically redacted from logs

---

## Data Security

| Measure | Detail |
|---|---|
| Soft deletes | Documents are never hard-deleted (audit trail preserved) |
| Audit logs | All auth actions logged with IP, user agent, timestamp |
| Audit log TTL | Auto-purged after 90 days (MongoDB TTL index) |
| Token TTLs | Refresh tokens, reset tokens, verification tokens auto-expire via TTL indexes |
| Optimistic concurrency | Prevents lost updates on concurrent writes (`__v` checks) |

---

## Dependency Security

- `npm audit` runs in CI on every push
- High-severity vulnerabilities block the security job (soft fail for moderate)
- All direct dependencies are pinned to exact versions in `package.json`

---

## Hardening Checklist (pre-production)

- [ ] `JWT_SECRET` is 64+ random hex characters (not the default)
- [ ] `CORS_ORIGIN` is set to your exact Vercel domain (not `*`)
- [ ] `MONGODB_URI` uses a dedicated database user with minimum required permissions
- [ ] Stripe webhook secret configured and all event types selected
- [ ] `NODE_ENV=production` set in Render
- [ ] `TRUST_PROXY=true` set in Render
- [ ] MongoDB Atlas network access restricted to Render's outbound IPs (or VPC peering)
- [ ] MongoDB Atlas backups enabled
- [ ] Rate limiting configured appropriately for expected traffic
